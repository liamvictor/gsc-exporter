GSC Exporter Automation Guide

A comprehensive, step-by-step guide to setting up automated, persistent Google Search Console caching and reporting tasks in Google Cloud Platform (GCP).

Why Can't We Use Google Cloud Shell?

While Google Cloud Shell is a fantastic environment for manual runs and configuration, it automatically shuts down after 20 to 30 minutes of user inactivity. When the container goes to sleep, any scheduled background cron jobs are halted.

To run caching and reporting tasks automatically (such as every week or month), you must run them on a persistent platform. Below are two fully documented, idiot-proof options.

Option 1: 24/7 VM

This is the simplest option. It runs a full virtual machine in the cloud that stays active 24/7. It is 100% free if you select the E2-micro VM size which qualifies for the GCP Free Tier.

Open Compute Engine Console
  1. Create the VM Instance Log into the Google Cloud Console and click "Create Instance". Name your VM (e.g., gsc-exporter-vm).
  2. Configure Free Tier Hardware Choose one of these US Regions: us-central1 (Iowa), us-east1 (S. Carolina), or us-west1 (Oregon).
    Under Machine configuration, select E2 -> e2-micro. (This machine size has 1GB RAM, which is completely free).
  3. Set Boot Disk Keep the default Debian GNU/Linux boot disk. Ensure the disk size does not exceed 30 GB to stay inside the Free Tier limits.
  4. Configure API Access Scopes Under "Identity and API access", select Allow full access to all Cloud APIs. This enables easy API operations. Click Create.
  5. SSH Connect and Install Software Once the VM starts, click the SSH button next to its name. In the terminal window that opens, paste these setup commands: 
    sudo apt-get update
sudo apt-get install -y git python3-pip python3-venv
git clone https://github.com/liamvictor/gsc-exporter.git ~/gsc-exporter
cd ~/gsc-exporter
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
  6. Upload Authentication Tokens You must copy your authenticated config/token.json and config/client_secret.json files from your local workspace or Cloud Shell onto this VM. You can use the VM's SSH window "Upload File" tool to place them into the ~/gsc-exporter/config/ directory.
  7. Schedule with Cron Open your user crontab configuration editor: 
    crontab -e
    If prompted, select nano (option 1) as the editor. Scroll to the bottom and paste this cron rule to execute the automation script every Monday at 4:00 AM: 
    0 4 * * 1 /bin/bash /home/username/gsc-exporter/cloud-shell-automate.sh --non-interactive --run-reports >> /home/username/gsc-exporter/cache-run.log 2>&1
    (Make sure you replace username with your actual Linux user name). Save the file by pressing Ctrl+O, Enter, and exit with Ctrl+X.
Tip: The E2-micro VM running Debian will consume zero credits and run your scheduler script indefinitely. You can review execution details in the cache-run.log file.

Option 2: Cloud Run + Scheduler

This is a serverless option. The script is packaged into a container. It runs only when triggered and scales down to zero, costing nothing unless your run duration exceeds GCP free quotas.

Open Cloud Run Console
  1. Enable GCP Services In your Cloud Shell, run this command to enable the required Build, Registry, Run, and Scheduler APIs: 
    gcloud services enable \
  cloudbuild.googleapis.com \
  artifactregistry.googleapis.com \
  run.googleapis.com \
  cloudscheduler.googleapis.com
  2. Create a Dockerfile Create a file named Dockerfile in your project root with these exact contents: 
    FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
ENTRYPOINT ["/bin/bash", "./cloud-shell-automate.sh", "--non-interactive", "--run-reports"]
  3. Build Container Image Submit a build request to Google Cloud Build to create your Docker container and store it in Artifact Registry: 
    gcloud builds submit --tag gcr.io/YOUR_PROJECT_ID/gsc-exporter
    (Replace YOUR_PROJECT_ID with your active Google Cloud project ID).
  4. Deploy a Cloud Run Job Go to the Cloud Run Console, click "Create Job" (do not select service, select Job because this is a background script, not a web server). Select your built container image. Set the timeout to 1800 seconds to allow large properties to cache.
  5. Create Cloud Scheduler Trigger Go to Cloud Scheduler Console and click "Create Job".
    • Frequency: 0 4 * * 1 (Every Monday at 4:00 AM)
    • Target type: HTTP
    • URL: The Cloud Run Job execution URL (from your Job settings page)
    • HTTP Method: POST
    • Auth Header: Add OIDC Token and select a service account with Cloud Run Invoker permissions.
Important: Keep your client_secret.json and token.json packaged securely inside the built container image. Ensure the Artifact Registry repository remains private.
