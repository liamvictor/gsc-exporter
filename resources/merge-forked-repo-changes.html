Git Workflow Guide

How to contribute changes from your fork back to the parent (upstream) repository efficiently.

Step 0: Setup Upstream Remote

Before you can sync, you must tell Git where the original repository is. You only need to do this once.

# Add the parent repo as 'upstream'
git remote add upstream https://github.com/liamdelahunty/gsc-exporter.git

Step 1: Push to your Fork

First, send your local commits to your own GitHub repository (the "origin").

git push origin main

Step 2: Create a Pull Request (PR)

The "Parent" repository (the original repo you forked from) is called the Upstream. To send your changes there:

  1. Go to your fork's page on GitHub: github.com/liamvictor/gsc-exporter.
  2. You should see a yellow bar saying "This branch is ahead of liamdelahunty:main".
  3. Click Contribute > Open Pull Request.
  4. Review your changes and click Create Pull Request.

Step 3: Keep your Fork in Sync (Rebase)

To avoid messy "Merge branch..." commits and keep your history clean, use rebase instead of merge when updating from the parent repo.

# 1. Fetch the latest changes from the parent repo
git fetch upstream

# 2. Rebase your local changes on top of the parent's latest code
# This moves your commits to the 'end' of the history
git rebase upstream/main

# 3. Force push to your fork (since history has been rewritten)
git push origin main --force

Best Practices for Upstream PRs

  • Squash and Merge: We recommend the maintainer uses "Squash and Merge" for Pull Requests. This condenses multiple incremental commits into a single clean feature commit on the main branch.
  • Avoid "Merge" Commits: Do not use the GitHub "Sync fork" button if it creates a merge commit. Use the rebase commands above instead to maintain a linear history.
  • Clean Commits: Group related changes into a single commit with a descriptive message. Use git commit --amend if you need to fix a small mistake in your last commit before pushing.
Pro Tip: Using rebase ensures that your Pull Request always looks like it was written against the very latest version of the project, making it much easier for maintainers to review and merge.
