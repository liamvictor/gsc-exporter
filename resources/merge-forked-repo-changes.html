Merging Forked Repository Changes

This document outlines the process for incorporating changes from a forked (or copied) repository into your "parent" repository using a Pull Request (PR).

Step 1: You Push Your Changes First

Since your local repository is ahead of your main branch on GitHub, you should push your changes first. This ensures the other user can base their work on the absolute latest version of your code.

  1. From your local repository, run: 
    git push origin main

Step 2: Instructions for the Other User

The other user needs to update their branch with your latest changes and then open the Pull Request. Please provide them with the following instructions.

  1. Add your repository as a remote. This is a one-time setup that creates a reference to your "parent" repository. They can name it upstream. They must replace <your_repo_url> with the actual URL of your GitHub repository. 
    git remote add upstream <your_repo_url>
  2. Fetch the latest changes from your repository. This will download the latest commits from your main branch without merging them yet. 
    git fetch upstream
  3. Rebase their branch. This is the most important step. It will take their local commits and re-apply them on top of the latest version of your main branch. This creates a clean, linear history and makes the final merge much simpler. 
    git rebase upstream/main

    Note: If they encounter any merge conflicts during the rebase, Git will pause and ask them to resolve the conflicts before continuing.

  4. Push the rebased branch to their fork. Because the history of their main branch was rewritten by the rebase, they will need to perform a "force push". Using --force-with-lease is a safer alternative to --force. 
    git push origin main --force-with-lease
  5. Create the Pull Request. They should now go to your repository's page on GitHub. GitHub will automatically detect that they pushed a new branch and will show a prominent "Compare & pull request" button. They should click it, review the changes, and submit the PR, ensuring the target is your main branch.

Step 3: You Review and Merge the Pull Request

Once the other user has created the Pull Request, you will get a notification.

  1. Go to the "Pull requests" tab in your GitHub repository.
  2. Click on the new PR to review the proposed changes, read their comments, and see a diff of all the code they want to merge.
  3. If everything looks good, click the "Merge pull request" button.
  4. Finally, pull the newly merged changes down to your local machine so it's fully up-to-date. 
    git pull origin main

This workflow is the standard for collaboration on GitHub and ensures that you have full control over what gets added to your repository.
