Google Search Console API Ideas

Account Wide Report

Github: gsc-exporter.

  • Device Performance Breakdown

    • Gather clicks, impressions, CTR, and average position segmented by device type (desktop, mobile, tablet) for each site. This would reveal cross-site device-specific trends.

  • Country Performance Breakdown

    • Include performance metrics broken down by the top X countries for each site. This could highlight geographic strengths and weaknesses across the account.

  • Top Queries/Pages

    • For each site, identify and list the top N queries and/or top N pages based on clicks or impressions. This would offer a snapshot of key content driving traffic across the portfolio.

  • Month-over-Month/Yea

    • over-Year Comparisons: Add columns or sections indicating percentage changes in clicks, impressions, CTR, and position compared to the previous month or the same month in the previous year.

  • Search Appearance (Ric

    • Results): Track and report on the presence and performance of different rich result types (e.g., FAQs, How-to, Product snippets) for each site.

  • Property Type and Verificatio

    • Status: Include metadata for each site, such as whether it's a URL-prefix or Domain property, and its verification status within GSC. This provides useful context.

  • Error/Exclusion Tracking

    • Summarize any significant GSC errors or excluded pages for each property (e.g., number of crawled but not indexed pages, server errors)

  • Average Query Position

    • Distribution: Instead of just average position, show the distribution of queries by position ranges (e.g., top 3, 4-10, 11-20, etc.) for each site.

  • For performance-analysis.py

    • This script's API request is hardcoded to the page dimension, so only page-based filters are appropriate without changing the core script logic.

    • `--page-contains <substring>`:
      • Use Case: Analyse performance changes just for pages within a specific section of your site, like /blog/ or /products/.
      • Example: py performance-analysis.py <site> --last-28-days --page-contains "/blog/"
    • `--page-exact <url>`:
      • Use Case: Get a performance comparison for a single, specific URL.
      • Example: py performance-analysis.py <site> --last-7-days --page-exact "https://example.com/my-article"
    • `--page-not-contains <substring>`:
      • Use Case: Exclude a section of your site from the analysis, for example, to see performance excluding /login/ pages.
      • Example: py performance-analysis.py <site> --last-month --page-not-contains "/login/"

Note: To filter by query in this script, its logic would need to be updated to also request the query dimension from the API.

---
  • For gsc-pages-queries.py & snapshot-report.py
  • These scripts fetch (or could fetch) both page and query dimensions, making them ideal candidates for a wide range of filters.
  • Query-based Filters:
    • * `--query-contains <substring>`:
      • * Use Case: Focus the report on a specific topic, keyword group, or brand name.
      • * Example: py gsc-pages-queries.py <site> --last-28-days --query-contains "brand name"
    • * `--query-exact <phrase>`:
      • * Use Case: Analyse performance for a single, exact-match keyword.
      • * Example: py gsc-pages-queries.py <site> --last-28-days --query-exact "how to bake bread"
    • * `--query-not-contains <substring>`:
      • * Use Case: The most common use is to exclude brand terms to analyse purely non-branded/organic discovery traffic.
      • * Example: py gsc-pages-queries.py <site> --last-28-days --query-not-contains "brand name"
    • Page-based Filters (same as above):
      • * --page-contains <substring>
      • * --page-exact <url>
      • * --page-not-contains <substring>
